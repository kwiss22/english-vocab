## 📊 옵션 비교

### 옵션 1: 별도 저장소로 분리 (권장 ⭐)

#### 구조:
```
loveallprince/loveall-tennis (테니스장 플랫폼)
loveallprince/loveall-vocab (영어 단어장)
```

#### 장점:
- ✅ **명확한 분리**: 각 프로젝트가 독립적
- ✅ **배포 간단**: Railway에서 각각 별도로 배포
- ✅ **관리 용이**: 프로젝트별로 독립적인 이슈/PR 관리
- ✅ **버전 관리 명확**: 각 프로젝트의 히스토리 분리
- ✅ **의존성 분리**: 각 프로젝트의 requirements.txt 분리

#### 단점:
- ❌ 저장소가 2개 필요

### 옵션 2: 같은 저장소, 서브디렉토리로 분리

#### 구조:
```
loveallprince/loveall/
├── tennis-platform/     (테니스장 플랫폼)
│   ├── package.json
│   ├── next.config.js
│   └── ...
└── vocab-app/           (영어 단어장)
    ├── web_vocab_app.py
    ├── Dockerfile
    ├── requirements.txt
    └── ...
```

#### 장점:
- ✅ 하나의 저장소로 관리
- ✅ 관련 프로젝트를 함께 볼 수 있음

#### 단점:
- ❌ **Railway 설정 복잡**: 어떤 디렉토리를 배포할지 설정 필요
- ❌ **Root Directory 설정**: Railway에서 각 서비스마다 Root Directory 지정 필요
- ❌ **배포 충돌 가능**: 두 프로젝트가 동시에 배포되면 혼란
- ❌ **관리 복잡**: 이슈/PR이 섞일 수 있음
- ❌ **의존성 충돌**: 두 프로젝트의 의존성이 다를 수 있음

### 옵션 3: 같은 저장소, 루트에 함께 (비권장 ❌)

#### 구조:
```
loveallprince/loveall/
├── package.json         (테니스장 플랫폼)
├── web_vocab_app.py    (영어 단어장)
```

#### 단점:
- ❌ **파일 충돌**: 같은 이름의 파일이 있을 수 있음
- ❌ **배포 불가능**: Railway가 어떤 프로젝트를 배포해야 할지 모름
- ❌ **혼란**: 프로젝트 구조가 복잡해짐
- ❌ **의존성 충돌**: package.json과 requirements.txt가 함께 있으면 혼란

## ✅ 권장 방법: 별도 저장소로 분리

### 방법 1: 새 저장소 생성 (가장 간단)

1. **새 GitHub 저장소 생성**
   - 저장소명: `loveallprince/english-vocab` 또는 `loveallprince/vocab-app`
   - 설명: "영어 단어장 웹 애플리케이션"

2. **로컬에서 새 저장소로 푸시**
   ```bash
   # 현재 디렉토리에서
   git remote add vocab-repo https://github.com/loveallprince/english-vocab.git
   git push vocab-repo main
   ```

3. **Railway에서 새 저장소 연결**
   - Settings → Source
   - 새 저장소 선택: `loveallprince/english-vocab`

---

### 방법 2: 기존 저장소에 추가 (서브디렉토리)

만약 같은 저장소에 두고 싶다면:

1. **GitHub에서 디렉토리 구조 생성**
   ```
   loveallprince/loveall/
   ├── tennis-platform/     (기존 Next.js 프로젝트)
   └── vocab-app/           (새로 추가할 영어 단어장)
   ```

2. **Railway 설정**
   - Settings → Source → Root Directory: `vocab-app`
   - 또는 새 서비스 생성하여 각각 다른 Root Directory 설정

3. **주의사항**
   - 각 서비스마다 Root Directory를 올바르게 설정해야 함
   - 배포 시 혼란을 피하기 위해 명확한 네이밍 필요
